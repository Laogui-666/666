沐海旅行 - 签证行业专属ERP系统详细设计与实施蓝图 (100% 可执行开发文档)
================================================================================
文档版本：V2.0 (终极详尽版 - 包含全工作流无遗漏拆解)
生成日期：2026-03-16
技术栈核心：Next.js (App Router) + React 18 + Prisma ORM + PostgreSQL + Tailwind CSS + Zustand + Socket.io

================================================================================
第一部分：项目可行性评估 (Feasibility Assessment)
================================================================================
1. 技术可行性：极高。
   - Next.js 的 Server Actions 和 API Routes 完美支持复杂的表单提交和状态流转。
   - Prisma ORM 配合 PostgreSQL，能够通过严格的外键约束和事务（Transactions）保证订单状态流转的原子性和一致性。
   - 多租户（Multi-tenancy）架构在数据库层面通过强制 `companyId` 过滤即可实现绝对的数据隔离。

2. 业务可行性：极高。
   - 签证行业的痛点在于“资料繁琐、流转节点多、跨部门沟通成本高”。本系统通过严格的状态机（State Machine）将非标服务标准化。
   - 预留的网店 API 接口和自动化短信/站内信通知，能大幅降低客服和资料员的沟通成本。

3. 风险提示与应对方案：
   - 风险：数据越权访问（如A公司操作员看到B公司数据）。
   - 应对：在后端封装统一的数据库查询中间件，所有查询必须带入当前登录用户的 `companyId`，从底层杜绝越权。

================================================================================
第二部分：系统架构设计 (System Architecture Design)
================================================================================
1. 核心模块划分 (模块化架构)：
   - IAM 模块 (Identity & Access Management)：负责登录、注册、JWT 鉴权、6级角色权限控制。
   - Tenant 模块 (多租户管理)：负责公司入驻、公司配置、数据隔离。
   - Workflow 模块 (核心工作流引擎)：负责订单状态机的流转、节点校验。
   - Document 模块 (资料与附件管理)：对接阿里云 OSS / AWS S3，负责文件的上传、预览、下载、归档。
   - Notification 模块 (消息中心)：负责站内信（WebSocket/SSE）、短信（第三方API）的触发与发送。

2. 数据库核心 Schema 设计 (Prisma 模型)：
   - Company: id, name, createdAt
   - Department: id, name, companyId
   - User: id, role, companyId, departmentId, email, passwordHash
   - Customer: id, name, phone, passportNo, userId (关联登录账号)
   - Order: id, companyId, customerId, csId(客服), dcId(资料员), opId(操作员), status, appointmentDate, createdAt, updatedAt
   - DocumentRequirement: id, orderId, name, isRequired, status (待传/待审/合格/打回), rejectReason, fileUrl
   - VisaMaterial: id, orderId, name, fileUrl, status (待审/合格/打回), rejectReason
   - OrderLog: id, orderId, userId, action, fromStatus, toStatus, timestamp (用于追溯每一个操作)

第三部分：账号系统与权限体系 (6级角色 RBAC 设计)
================================================================================
1. 超级管理员 (SUPER_ADMIN)
   - 权限：本网站最高权限。管理所有入驻公司（SaaS租户），配置系统全局参数。

2. 公司负责人 (COMPANY_OWNER)
   - 权限：该公司数据范围内账号最高权限。
   - 范围：可以使用管理系统中的该公司的所有功能，查阅该公司数据库下所有数据（包括对公司名下所有子账号的删改查和数据调阅）。公司之间数据绝对隔离。

3. 业务部管理员 (DEPT_ADMIN)
   - 权限：部门级最高权限。
   - 范围：可以使用该公司管理系统中该部门的相应功能，查阅该部门汇总的数据和该部门所有子账号的数据。

4. 业务部门操作员/资料员 (OPERATOR / DOC_COLLECTOR)
   - 权限：员工级权限。
   - 范围：可以使用该公司管理系统中该部门的相应功能，仅能查阅和操作自己名下的数据（分配给自己的订单）。

5. 外包业务员 (OUTSOURCE)
   - 权限：受限员工级权限。
   - 范围：可以使用该公司管理系统中经授权的相应功能，仅能查阅自己名下的数据。敏感客户信息（如完整联系方式）可配置为脱敏显示。

6. 普通用户 (CUSTOMER)
   - 权限：客户端 End-User 权限。
   - 范围：只能从个人信息面板中的“我的订单”进入订单管理页面进行交互（上传资料、查看进度）。

第四部分：完整详细无遗漏的全工作流拆解 (Complete Workflow Breakdown)
================================================================================
系统核心状态机枚举 (OrderStatus):
PENDING_CONNECTION(待对接) -> CONNECTED(已对接) -> COLLECTING_DOCS(资料收集中) -> PENDING_REVIEW(待审核) -> UNDER_REVIEW(资料审核中) -> MAKING_MATERIALS(材料制作中) -> PENDING_DELIVERY(待交付) -> DELIVERED(已交付) -> APPROVED/REJECTED(出签/拒签)

【一、 客服接单录单】
1. 动作：客服将已下单付款客户的订单信息录入系统（预留网店API同步接口）。核对无误后点击“提交”。
2. 系统底层逻辑与流转：
   - 数据库创建 Order 记录，初始状态设为 `PENDING_CONNECTION` (待对接)。
   - 订单出现在客服的“已接单”界面，同时推送到资料部的“待接单”公共池。
   - 账号判定：通过填入的客户信息（手机号等），系统自动为客户创建账号。若已有账号则直接关联。
   - 订单同步推送到客户账号的客户端“订单列表”。
   - 触发 WebSocket 广播：资料员界面弹出实时通知：“又有新订单啦”。
   - 触发 SMS 接口：自动向客户发送短信提醒。
     * 短信模板：“感谢您的信任，资料员将在24小时内与您取得联系，您可以通过平台地址：www.Xxxxxx.com ，登陆账号：[账号]，密码：[密码]，查看您的订单状态及反馈。盼达旅行竭诚为您服务！”

【二、 资料员接单及收集反馈资料】
1. 动作：资料员在“待接单”界面选择订单，点击“接单”。
   - 系统逻辑：订单绑定该资料员(`dcId`)，从公共池移入该资料员的“已接单”界面。
   - 状态变更：`CONNECTED` (已对接)。(注：资料员需联系客户告知在平台或企微进行对接)。
2. 动作：资料员在详情页补充/修改/备注客户信息，点击“资料收集”弹出面板。选择或手动添加所需资料清单，点击“发送客户”。
   - 系统逻辑：生成 DocumentRequirement 记录。
   - 状态变更：`COLLECTING_DOCS` (资料收集中)。
   - 触发通知：客户界面订单状态更新，收到系统通知提醒。
3. 动作：客户在客户端订单详情的“资料收集面板”中上传资料或备注回复，点击“确认提交”。
   - 系统逻辑：更新 DocumentRequirement 的文件URL和状态。
   - 触发通知：资料员界面订单状态更新，收到系统通知提醒。
4. 动作：资料员审核客户上传的资料（预览/下载）。
   - 合格：打勾。
   - 不合格：备注修改建议。
   - 补充：手动添加新资料项。
   - 最终点击“发送客户”。
   - 系统逻辑：更新对应资料的状态和驳回原因。
   - 触发通知：客户界面订单状态更新，收到系统通知提醒。
5. 循环：重复步骤 3 和 4，直到资料员认为资料达到送签标准。
6. 动作：资料员点击“提交审核”。
   - 系统逻辑：订单推送到操作部的“待接单”公共池。
   - 状态变更：`PENDING_REVIEW` (待审核)。
   - 触发通知：操作员界面弹出实时通知：“又有新订单啦”。自动向客户发送短信提醒。
     * 短信模板：“感谢您的配合与支持，您的材料已提交送签专员复审啦，您可以通过平台地址：www.Xxxxxx.com ，登陆账号：[账号]，密码：[密码]，查看您的订单状态及反馈。盼达旅行竭诚为您服务！”

【三、 操作员接单及审核反馈】
1. 动作：操作员在“待接单”界面选择订单，点击“接单”。
   - 系统逻辑：订单绑定该操作员(`opId`)，移入该操作员的“已接单”界面。
   - 状态变更：`UNDER_REVIEW` (资料审核中)。
2. 动作：操作员审核资料。合格打勾，不合格备注建议，或添加需补充资料。点击“反馈资料员”。
   - 系统逻辑：记录审核意见。
   - 触发通知：资料员界面订单状态更新，收到系统通知提醒。
3. 动作：资料员查看操作员反馈。
   - 分支 A：资料员自己直接补充上传资料或回复疑问。
   - 分支 B：需客户补充，资料员点击“发送客户”。
     - 系统逻辑：状态回退至 `COLLECTING_DOCS` (资料收集中)。触发客户通知。
4. 循环：重复资料收集的交互，直到资料员认为达标，点击“提交复审”。
   - 系统逻辑：状态变更为 `UNDER_REVIEW` (资料审核中)。触发操作员通知。
5. 动作：操作员认为资料已达标，点击“开始制作签证材料”。
   - 系统逻辑：状态变更为 `MAKING_MATERIALS` (材料制作中)。
   - 触发通知：客户、资料员界面订单状态更新，并收到系统通知提醒。

【四、 操作员签证材料制作与交付】
1. 动作：操作员制作完成后，在“签证材料面板”上传资料包或单个文件，点击“发送资料”。
   - 系统逻辑：生成 VisaMaterial 记录。
   - 状态变更：`PENDING_DELIVERY` (待交付)。
   - 触发通知：资料员界面订单状态更新，收到系统通知提醒。
2. 动作：资料员查看送签资料。若需补充/修改，在面板中填写反馈，点击“反馈操作员”。
   - 系统逻辑：记录反馈意见。
   - 状态变更：回退至 `MAKING_MATERIALS` (材料制作中)。
   - 触发通知：操作员界面订单状态更新，收到系统通知提醒。
3. 动作：操作员查看反馈，重新上传修改后的材料/备注，点击“发送资料”。
   - 系统逻辑：状态再次变更为 `PENDING_DELIVERY` (待交付)。触发资料员通知。
4. 循环：重复步骤 2 和 3，直到资料员认为送签资料达到交付标准。
5. 动作：资料员在“签证材料面板”点击“材料交付”。
   - 系统逻辑：状态变更为 `DELIVERED` (已交付)。
   - 触发通知：客户界面订单状态更新，收到系统通知提醒。自动向客户发送短信提醒。
     * 短信模板：“久等啦！您的签证材料已制作完成，您可以通过平台地址：www.Xxxxxx.com ，登陆账号：[账号]，密码：[密码]，查看您的订单状态及下载送签资料。祝您签证顺利！”

【五、 客户送签及签证结果反馈】
1. 动作：系统自动监控需要录指纹/面试的订单。
   - 系统逻辑：定时任务 (Cron Job) 每天运行，匹配预约日期 `appointmentDate`。
   - 触发通知：预约前一天自动发送短信提醒。
     * 短信模板：“您预约的[YYYY]年[MM]月[DD]日[HH:MM] 签证中心录指纹及递交签证材料，请别忘了准备好相关材料，安排好出行时间噢。若递交材料时遇到任何问题，请直接与我们联系。祝您签证顺利！”
2. 动作：出签状态反馈。
   - 申根签证：客户在客户端反馈，或资料员手动修改。
   - 电子签证：操作员反馈提交。
   - 点击“提交”签证结果。
   - 系统逻辑：状态变更为 `APPROVED` (出签) 或 `REJECTED` (拒签)。
   - 触发通知：客户界面订单状态更新，收到系统通知提醒。
3. 动作：发送结果短信。
   - 员工点选相应短信内容模板，点击“发送短信通知”。
   - 系统逻辑：调用 SMS 接口向客户发送最终出签/拒签短信。流程彻底闭环。

第五部分：工作流底层实现逻辑与技术方案 (Workflow Implementation Logic)
================================================================================
为确保“每一个流程，每一个交互的严谨”，且“不会一个bug牵一发而动全身”，系统必须采用以下底层设计：

1. 状态机模式 (State Machine Pattern)
   - 严禁在代码中随意修改 `status` 字段。必须封装统一的 `TransitionService`。
   - 每次状态变更必须校验：当前状态是否允许向目标状态流转？操作人是否拥有权限？
   - 示例：只有状态为 `COLLECTING_DOCS` 且当前用户是该订单的 `dcId` 时，才能触发“提交审核”流转到 `PENDING_REVIEW`。

2. 数据库事务 (Database Transactions)
   - 状态变更、写入操作日志 (OrderLog)、生成通知 (Notification) 必须放在同一个 Prisma 事务中。
   - 确保原子性：要么全部成功，要么全部失败。绝不出现“状态改了但没发通知”的 Bug。

3. 事件驱动架构 (Event-Driven Architecture)
   - 状态变更成功后，触发内部事件（如 `ORDER_STATUS_CHANGED`）。
   - 独立的监听器 (Listeners) 监听到事件后，异步执行非核心阻塞任务：如调用阿里云发短信、通过 Socket.io 推送前端弹窗。这保证了主流程的极速响应。

4. 模块化组件 (Modular Components)
   - 前端将“资料收集面板”、“签证材料面板”、“订单时间轴”封装为独立的 React Components。
   - 根据当前订单的 `status` 和当前登录用户的 `role`，动态决定组件是“只读模式”、“编辑模式”还是“隐藏模式”。

第六部分：各权限等级 UI 界面设计方案 (UI/UX Design Scheme)
================================================================================

公共布局 (Layout):
- 左侧：可折叠的深色侧边栏 (Sidebar)，包含导航菜单。
- 顶部：顶栏 (Topbar)，包含全局搜索、面包屑导航、消息通知铃铛、个人中心下拉菜单。
- 主体：浅灰色背景的内容区 (Main Content)。

1. 客户端 UI (Customer Portal)
- 风格：移动端优先，清晰、温和、减少焦虑感。
- 首页：大卡片展示当前进行中的订单。
- 订单详情页：
  - 顶部：横向步骤条 (Timeline)，高亮当前进度。
  - 核心区：【资料收集面板】。列表显示需要上传的材料。
    - 状态图标：灰色(待上传)、蓝色(审核中)、绿色(已合格)、红色(需修改)。
    - 交互：点击列表项弹出底部抽屉，支持拍照/相册/文件上传。显示资料员的修改建议。
  - 底部：【签证材料面板】（仅在“已交付”状态后解锁），提供下载按钮。

2. 资料员/操作员 UI (Operator Dashboard)
- 风格：桌面端优先，高密度信息，强调操作效率。
- 工作台首页：
  - 顶部：数据概览（今日待办、处理中、被打回）。
  - 主体：看板视图 (Kanban Board)。列名对应订单状态。支持拖拽（如将订单从“待审核”拖入“审核中”即完成接单）。
- 订单处理详情页 (分屏设计 Split-pane)：
  - 左侧面板 (30%宽度)：客户基本信息、订单日志时间轴、内部沟通备注区。
  - 右侧面板 (70%宽度)：
    - 【资料收集/审核面板】：左侧显示文件列表，右侧为 PDF/图片内嵌预览器。
    - 底部操作栏：悬浮的“全部合格并提交”、“打回修改”按钮。

3. 业务部管理员 UI (Department Admin)
- 风格：数据监控与人员调度。
- 部门看板：
  - 图表区：折线图展示本周接单量趋势，饼图展示各状态订单占比。
  - 人员负荷表：列表展示部门内每个员工的当前处理量。
  - 异常监控区：高亮显示“停留同一状态超过 48 小时”的订单。
- 交互：在人员负荷表中点击某订单，可强制执行“转单”操作，重新分配 `dcId` 或 `opId`。

4. 公司负责人 UI (Company Owner)
- 风格：宏观视角，企业管理。
- 驾驶舱 (Dashboard)：
  - 核心指标：总营收、总订单量、整体出签率、各部门绩效排名。
- 组织架构管理页：
  - 树状图展示部门结构。
  - 员工列表：支持增删改查，分配角色（DEPT_ADMIN, OPERATOR 等）。
- 数据穿透：点击任何统计数字，可下钻查看具体的订单列表。

5. 超级管理员 UI (Super Admin)
- 风格：SaaS 平台后台。
- 租户管理页：
  - 列表展示所有入驻公司。显示公司名称、负责人、联系方式、入驻时间、账号状态（启用/封禁）。
- 系统设置页：
  - 短信模板配置、系统全局日志审计。

6. 外包业务员 UI (Outsource Worker)
- 风格：极简，受限视图。
- 界面类似操作员，但：
  - 隐藏客户敏感信息（手机号、护照号部分打码）。
  - 只能看到分配给自己的订单，无权查看部门其他数据。
  - 无法导出客户原始资料，只能在线预览。

第七部分：实施方案与开发路线图 (Implementation Roadmap)
================================================================================
Phase 1: 基础设施与权限基座 (Week 1-2)
- 初始化 Next.js 项目，配置 Tailwind CSS 和 Shadcn UI。
- 设计并实例化 Prisma Schema，运行数据库迁移。
- 实现 NextAuth.js 登录，封装带 `companyId` 校验的鉴权中间件 (Middleware)。

Phase 2: 核心工作流引擎闭环 (Week 3-5)
- 开发各角色的基础 UI 框架 (Sidebar, Topbar)。
- 实现订单的 CRUD 接口。
